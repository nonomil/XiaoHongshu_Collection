# xhs-cdp-mainline-ops

## 目标

在不推翻当前架构的前提下，优先把现有 CDP 主线做稳。

默认立场：
- 优先修现有链路
- 优先增加诊断与反馈
- 允许人工接管
- 不默认迁移到 Playwright

## 问题分层

先把问题归到一层，再行动：

| 层级 | 现象 | 优先动作 |
|------|------|---------|
| 浏览器接入 | 未检测到调试会话、9222 不通、current-browser 失联 | 先修接入与模式切换 |
| 页面访问 | 笔记详情打不开、跳 404、要登录 | 先修访问方式与登录态 |
| 评论加载 | 页面评论存在但抓到 0 或很少 | 先查等待、滚动、虚拟列表 |
| 评论权限 | 登录查看全部评论、406、-101、300011 | 明确提示门槛/风控/账号异常 |
| 表达层 | 实际抓到了，但用户看不懂为什么失败 | 优先补 UI 提示和结果说明 |

## 默认工作流

### A. 浏览器接入失败

1. 先确认当前配置是 `current-browser` 还是 `isolated`
2. 若 `current-browser` 失败：
   - 优先提供切换到项目浏览器的一键修复
   - 再引导用户检查当前 Chrome 远程调试
3. 若 `isolated` 失败：
   - 检查项目浏览器是否成功拉起
   - 检查 profile、调试端口、登录态

### B. 评论抓取不完整

1. 先检查评论容器是否真正 ready
2. 再区分：
   - 懒加载未完成
   - 顶层评论还没滚完
   - 虚拟列表窗口覆盖
   - 网页端登录门槛
   - 评论接口受限或账号异常
3. 若属于加载问题：
   - 优先改等待、滚动、快照累积器
4. 若属于权限或风控：
   - 不承诺自动抓全
   - 引导用户切到人工处理后继续

### C. UI 和提示优化

优先补这些：
- 失败层级
- 失败原因
- warning code
- 评论总数 / 实际抓取数
- 是否需要人工接管

## 建议优先落地的改进

### 1. 诊断能力

- 把失败层级直接展示给用户
- 把“接入失败 / 详情页失败 / 评论加载失败 / 评论接口受限 / 登录门槛”区分开
- 不要只给统一的失败文案

### 2. 评论可观测性

- 在 UI 里显眼展示 `commentWarningCode`
- 展示 `commentTotal`
- 展示实际抓到条数
- 展示是否因为登录门槛或风控提前停止

### 3. 人工接管点

- 对验证码、风控、登录查看全部评论，不要继续自动重试到底
- 明确停下
- 引导用户去当前浏览器处理
- 提供处理后继续执行入口

### 4. 具体落地项

- 浏览器接入失败的一键修复
- 评论抓取不完整的分层提示
- `commentTotal`、实际条数、`commentWarningCode` 的结果区展示
- 风控/登录门槛时的人工接管按钮

## 禁止事项

- 不要默认重构成 Playwright
- 不要把所有评论问题都归因到代码 bug
- 不要忽略登录门槛和风控的客观存在
- 不要在没有证据时做“继续自动重试一定会好”的承诺
