# 单笔记评论区采集设计
日期：2026-03-08

## 背景

当前单笔记导出已经覆盖正文、图片、OCR、摘要和标签，但评论区仍未进入归档链路。对小红书这类内容来说，评论区经常包含工具名、纠错、经验补充、替代方案和作者追答，信息密度并不低。

如果直接把全部评论原样拼进主 Markdown，会引入三个问题：

- 噪声太高，阅读体验很差
- 文件体积会明显膨胀
- 后续想重跑筛选和总结时必须重新抓取

因此评论区需要独立归档，并增加一层 AI 筛选和总结。

## 目标

- 采集当前笔记下全部可展开评论和楼中楼
- 将原始评论单独归档，避免污染主 Markdown
- 对评论做保守筛选，保留高价值补充信息
- 生成评论区 AI 总结，并输出有用评论全文
- 评论采集失败时不阻塞正文、图片、OCR 主流程

## 非目标

- 本轮不做评论情感分析
- 本轮不做评论点赞排序策略优化
- 本轮不做评论增量同步
- 本轮不做评论区图片、表情包媒体下载
- 本轮不追求一次性解决仓库全部编码污染问题

## 方案对比

### 方案 A：全部评论直接写进主 Markdown

优点：

- 实现最省事
- 不需要额外归档文件

缺点：

- 文件会非常长
- 噪声和废话难以控制
- 无法复用原始评论数据做二次筛选

结论：不推荐。

### 方案 B：原始评论单独归档，主 Markdown 只放总结和有用评论

优点：

- 主文件保持可读
- 原始数据可复用，可重跑 AI 过滤和总结
- 适合后续继续扩展结构化字段

缺点：

- 需要新增一个归档文件
- 导出链路比正文多一步

结论：推荐。

## 推荐设计

### 输出结构

单笔记导出完成后，默认输出两份结果：

- 主文件：`output/单挑保存/<标题>.md`
- 评论归档：`output/单挑保存/_comments/<noteId>.json`

主 Markdown 仅新增两段：

- `## 评论区总结`
- `## 有用评论全文`

原始评论不直接拼进主文件，而是写入 `_comments/<noteId>.json`。

### 评论数据结构

原始评论按结构化 JSON 保存，至少包含：

- `commentId`
- `parentId`
- `rootId`
- `author`
- `content`
- `date`
- `likeCount`
- `replyCount`
- `level`
- `isAuthor`

归档文件顶层建议包含：

- `noteId`
- `noteTitle`
- `collectedAt`
- `totalComments`
- `comments`

### 评论采集策略

默认采集全部评论和楼中楼，直到当前详情页可见评论全部展开完成。

抓取顺序：

1. 展开主评论列表
2. 展开每条评论下的“展开更多回复”
3. 逐条读取评论节点
4. 结构化去重

采集失败处理：

- 评论区失败不阻塞正文导出
- 主 Markdown 中标记“评论采集失败”
- 不删除旧的评论归档文件

### AI 处理链路

评论处理独立于正文 OCR 链路，顺序如下：

1. 原始评论清洗
2. 基础去重
3. AI 判断评论是否有用
4. 基于有用评论生成总结
5. 将有用评论全文写入主 Markdown

### 有用评论判定策略

默认采用保守筛选，宁可多留，不优先极限压缩。

优先保留：

- 工具名、产品名、网址、关键词
- 错误纠正、补充说明
- 使用经验、踩坑、反例
- 作者对问题的有效回复
- 对正文信息的关键补完

优先过滤：

- 纯情绪表达
- 无信息量夸赞
- 纯表情、玩梗、复读
- 只有“求链接”“蹲”等短句且没有后续有效回复

### Markdown 结构

在现有正文、OCR、图片区块后追加：

```md
## 评论区总结

<AI 总结文本>

## 有用评论全文

### 评论 1
- 作者：
- 时间：
- 点赞：
- 内容：

### 评论 2
...
```

如果没有筛出有用评论，则输出：

- `评论区已采集，但未筛出高价值评论`

如果评论采集失败，则输出：

- `评论区采集失败，本次仅导出正文与图片内容`

## 错误处理

需要明确区分：

- 页面未登录导致评论不可见
- 评论节点结构变化导致提取失败
- 展开更多评论失败
- AI 过滤失败
- AI 总结失败

处理原则：

- 采集失败不阻塞主导出
- AI 过滤失败时退化为规则筛选
- AI 总结失败时仍输出有用评论全文和原始归档

## 测试策略

优先覆盖纯函数和结构化输出：

- 评论结构清洗与去重
- 有用评论 Markdown 渲染
- 评论归档路径生成
- 评论数据写入 JSON
- 无评论、评论失败、有评论三种输出分支

浏览器评论展开逻辑本轮只做小范围集成验证，不做重型端到端自动化。

## 结论

评论区能力应作为单笔记导出的附加链路实现，但必须和主 Markdown 解耦。最稳妥的落地方式是“原始评论 JSON 归档 + 主文件只保留评论总结和有用评论全文”。这样既能保留全部信息，又不会牺牲主文件的可读性，也为后续重跑 AI 过滤和总结留出空间。
