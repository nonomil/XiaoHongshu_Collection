# 统一入站插件与云端接入设计

## 背景

当前项目已经具备稳定的本地归档内核：

- 任务对象已统一为 `task`
- 执行链路已统一为 `input -> fetch -> enrich -> write -> report`
- UI、CLI、Pushbullet 收件箱都已经在不同层面复用同一套保存能力

但日常使用仍有明显摩擦：

- 浏览器里想保存当前页时，需要先打开 `启动小红书保存入口.bat`
- 还需要手动复制链接或粘贴分享文本
- 手机端、飞书、Pushbullet、浏览器插件尚未统一接到一个入站口

用户目标已经明确：

1. 在浏览器里点一下插件，就能把当前页或当前链接送进现有归档管线
2. 后续项目部署到云端 VPS 后，插件也能连接云端服务
3. 手机分享、飞书机器人、Pushbullet 也能接入同一个保存入口

---

## 设计结论

### 结论 1：插件不是新内核，只是统一入口

Chrome 插件不负责抓取、OCR、摘要、分类和落盘，也不重写现有保存逻辑。

插件只负责：

- 获取当前标签页 URL
- 可选读取页面标题与选中文本
- 将结构化请求发送给本地或云端入站 API
- 展示简单状态：已发送、处理中、失败、打开工作台

这样可以最大化复用当前项目已经成熟的 Node/CDP/文件输出链路，避免把本地能力硬搬进 MV3 插件里。

### 结论 2：主工程先不迁移，新增 `prj/` 承接新产品形态

主仓库继续保留当前内核目录：

- `scripts/`
- `ui/`
- `config/`
- `data/`
- `output/`

新增：

```text
prj/
  README.md
  chrome-extension/
  desktop-exe/
```

其中：

- `prj/chrome-extension/` 是第一优先级
- `prj/desktop-exe/` 先占位，第二阶段再做

不在第一阶段大规模抽 `core/`，避免无收益的大搬家和回归风险。

### 结论 3：统一入站比“多入口各做一套”更重要

后续所有入口都应收敛到同一套入站协议：

- Chrome 插件
- 本地 UI
- Pushbullet
- 飞书机器人 / Webhook
- 手机分享桥接
- 云端监控或 OpenClaw 调度

真正的差异只体现在“输入来源”和“执行位置”，而不是保存逻辑本身。

## 总体架构

### 组件划分

#### 1. 浏览器侧入口

目录：`prj/chrome-extension/`

职责：

- 获取当前活动标签页 URL
- 允许用户选择发送目标：本地 / 云端
- 触发“保存当前页”
- 打开工作台或最近结果页

第一版不做复杂页面 DOM 提取，不做登录态桥接，不做直接内容抓取。

#### 2. 入站 API 层

归属：主工程 `scripts/` + `ui_server.js`

职责：

- 接收插件或外部调用的统一请求
- 校验来源、密钥和 payload
- 把请求转换为内部 `task`
- 决定立即执行还是写入队列

#### 3. 队列 / 收件箱层

归属：主工程 `data/` + `scripts/lib/inbox_*`

职责：

- 存储待处理 URL
- 去重
- 记录来源、时间、目标环境、本地/云端路由
- 支持后续手机、飞书、Pushbullet 共用

第一阶段可以先基于现有 JSONL inbox 扩展，而不是立刻引入数据库。

#### 4. 归档执行器

归属：现有保存链路

职责：

- 调用现有 `save_note` / `saveLinksText` / 收藏夹与 inbox 保存逻辑
- 输出 report
- 更新任务历史与结果

执行器保持现状，尽量少动。

## 数据流设计

### 场景 A：本地浏览器插件一键保存

1. 用户点击插件按钮“保存当前页到本地”
2. 插件读取当前标签页 URL、标题、来源页信息
3. 插件向本地服务发送 `POST /api/ingress/save-link`
4. 本地服务将 payload 规范化为 `note-save` 任务
5. 服务直接调用现有保存能力
6. 返回任务状态、摘要与输出位置

适用：

- 你自己的电脑
- 已存在项目专用浏览器登录态或当前浏览器复用能力
- 目标是替代复制粘贴和 `bat` 启动

### 场景 B：浏览器插件发送到云端队列

1. 用户点击“发送到云端”
2. 插件向 VPS 上的 ingress API 发送请求
3. 云端先写入队列或 inbox，不保证立刻抓取
4. Worker 再决定立即处理、延后处理、人工介入

适用：

- 外部电脑、临时设备
- 想先收集链接，稍后统一跑
- 手机和多端统一入口

### 场景 C：手机 / 飞书 / Pushbullet 自动入站

1. 外部入口把链接写入统一 inbox
2. inbox 记录来源，例如 `pushbullet`、`feishu`、`mobile-share`
3. 统一保存任务读取 inbox 并复用现有 `saveInboxUrls`
4. 落盘与分类逻辑仍由主工程处理

适用：

- 被动收集
- 自动化巡检
- 跨设备投递

## API 设计

### 第一阶段新增 API

#### `POST /api/ingress/save-link`

用途：

- 浏览器插件或本地工具直接提交一个 URL，并尝试立即执行保存

请求建议：

```json
{
  "url": "https://www.xiaohongshu.com/explore/abc123",
  "source": "chrome-extension",
  "route": "local",
  "pageTitle": "示例标题",
  "selectionText": "",
  "targetMode": "immediate"
}
```

响应建议：

```json
{
  "ok": true,
  "task": "note-save",
  "accepted": true,
  "execution": "immediate",
  "report": {}
}
```

#### `POST /api/ingress/enqueue-link`

用途：

- 只写入队列 / inbox，不立即执行

适用：

- 云端
- 手机分享
- 飞书 / Pushbullet / Webhook

### 认证策略

第一阶段分两种模式：

- 本地模式：仅监听 `127.0.0.1`，不额外暴露公网，靠本地端口隔离
- 云端模式：必须带 token，默认禁用未鉴权公网写入

第二阶段再补：

- 入口来源标识
- 细粒度 token
- 可选 IP 白名单

## 本地与云端边界

### 本地优先处理的内容

- 小红书单页保存
- 当前浏览器上下文相关的抓取
- 依赖现有本机 Chrome 或项目 profile 的保存

### 云端优先处理的内容

- 纯 URL 收集
- 微信文章、知乎、CSDN 这类更适合后处理的页面
- 批量任务、定时任务、自动化监控

### 暂不承诺的内容

第一阶段不承诺“云端立即完整抓取小红书内容并稳定拿到评论”。

原因：

- 小红书强依赖登录态
- 浏览器上下文、人工预热、验证码、评论风控都更适合本地或专用浏览器环境

因此云端对小红书的第一阶段策略应为：

- 先收链接
- 再决定是否由云端浏览器处理
- 或下发到本地执行器处理

## 插件产品设计

### 第一版最小功能

插件弹窗只提供 3 个主按钮：

1. 保存当前页到本地
2. 发送当前页到云端
3. 打开工作台 / 查看最近结果

### 第一版不做的内容

- 不做复杂设置面板
- 不做批量标签页扫描
- 不做页面内复杂采集脚本
- 不做本地浏览器会话修复
- 不做评论提取状态可视化

### 后续可扩展能力

- 右键菜单“发送当前链接到归档”
- 页面分享按钮
- 选中文本解析多链接
- 最近发送历史
- 本地 / 云端目标切换

---

## 队列与任务模型

现有 `task` 模型已经足以支持第一阶段，只需要补充元信息：

- `source`: `chrome-extension` / `pushbullet` / `feishu` / `mobile-share`
- `route`: `local` / `cloud`
- `delivery_mode`: `immediate` / `queue`
- `requested_at`
- `metadata.page_title`
- `metadata.selection_text`

第一阶段不需要改造为数据库任务系统，先在现有 JSONL / 文件型 inbox 基础上扩展字段即可。

## 错误处理

### 插件侧

只显示用户可理解的状态：

- 已发送
- 本地服务未启动
- 云端鉴权失败
- 任务已入队
- 保存失败，请到工作台查看详情

### 服务侧

保留详细 report 和 warning：

- 不支持的来源类型
- 浏览器未连接
- 登录态不可用
- 小红书需要人工介入
- 队列写入失败

### 降级策略

- 本地立即执行失败时，可选回退为“仅入队”
- 小红书抓取失败时，允许保留链接任务供后续重试
- 插件不直接承担复杂恢复逻辑

## 分阶段实施建议

### Phase 1：本地一键保存 MVP

目标：

- 替代 `bat + 复制粘贴链接`
- 在浏览器里一键送进本地归档服务

交付：

- `prj/chrome-extension/`
- 本地 `ingress` API
- 本地立即执行保存

### Phase 2：云端入站与队列

目标：

- 插件能把链接发到 VPS
- 手机、飞书、Pushbullet 都接入同一个队列

交付：

- 公网可配置 `ingress` API
- token 鉴权
- 统一 inbox/queue 扩展

### Phase 3：多端协同与 EXE 壳

目标：

- 桌面启动体验更轻
- 多端入口统一
- 本地与云端路由可配置

交付：

- `prj/desktop-exe/`
- 最近任务视图增强
- 路由与环境切换配置

## 设计选择对比

### 方案 A：直接把现有 UI 打包成插件

不选。

原因：

- 当前 UI 强依赖本地 Node 服务
- 现有抓取与落盘不适合直接运行在 MV3 插件内
- 迁移成本高，收益低

### 方案 B：插件做轻量入口，服务做执行器

选这个。

原因：

- 与当前架构最兼容
- 能最快解决“一键保存”
- 也最容易扩展到云端和多入口

### 方案 C：先做 EXE，再做插件

暂不优先。

原因：

- EXE 解决的是启动体验
- 插件解决的是浏览器内最频繁的操作摩擦
- 当前用户痛点更偏向“入口触发”而不是“桌面包装”

---

## 最终决策

最终采用：

- 引入 `prj/`
- 优先建设 `prj/chrome-extension/`
- 插件定位为统一入口，不承载核心抓取逻辑
- 主工程新增统一入站 API
- 本地优先，云端第二阶段接入
- 手机 / 飞书 / Pushbullet 统一对接一个 inbox/queue

这条路线能最小化对现有保存能力的扰动，同时最大化改善“触发保存”的体验。
