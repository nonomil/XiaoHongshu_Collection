# 多平台扩展方向评估

日期：2026-03-12

## 问题

当前项目已经聚焦小红书收藏夹、单笔记保存、OCR、评论提取和 Markdown 归档。

新增问题是：项目是否值得继续扩展到以下内容平台：

- 微信公众号文章
- 知乎文章
- CSDN 博客
- 其他类似“网页文章型”内容平台

核心判断标准不是“能不能抓”，而是：

1. 是否显著提升用户价值
2. 是否会稀释当前项目优势
3. 是否适合当前项目的架构演进方向

## 调研结论

### 1. 微信公众号文章：值得支持正文保存，不建议承诺评论

从当前公开资料看，微信公众号文章的“正文保存”已有不少现成实践：

- `WeChat Article to Markdown` Chrome 扩展明确主打“将微信公众号文章内容提取为 Markdown”，并说明同时兼容博客园、掘金、CSDN、GitHub 等通用技术博客。
- `wechat-article-exporter` GitHub 组织下的 `wechat-article-exporter` 明确描述为“微信公众号文章批量下载工具”，并宣称支持导出阅读量和评论数据。
- `zhihu-download` 项目更新日志显示它后来也加入了“微信公众号文章下载功能”和“CSDN 博客文章下载功能”。

但评论能力上，微信更适合保守判断：

- 我没有找到稳定、官方、公开的“网页端评论采集接口”说明。
- 当前公开的近期信息更多是在描述微信 App 内的评论和互动能力增强，例如图片评论、长按文字评论等。
- 这意味着：即使技术上有人做了评论导出，也更可能依赖非官方页面结构、登录态或私有接口，长期稳定性和合规性都不适合作为当前项目的承诺功能。

结论：

- 微信公众号文章正文保存：值得做
- 微信公众号评论提取：不要作为一期承诺能力

### 2. 知乎文章：很适合纳入

知乎是最适合纳入的扩展目标之一。

原因：

- 文章页是标准网页内容，结构比小红书更稳定
- 文字、图片、标题、作者、发布时间、专栏信息更适合 Markdown 落地
- 已有现成开源工具 `zhihu-download`，目标就是把知乎专栏文章转 Markdown
- 也已经存在面向 Obsidian 的知乎插件生态，说明“知乎 -> 本地知识库”这个链路有明确需求

评论方面：

- 网页端评论在知乎上通常比微信公众号更可见
- 但当前项目如果扩平台，一期仍不建议把“评论”作为所有平台都要补齐的基线功能

结论：

- 知乎正文归档非常值得做
- 评论可以延后

### 3. CSDN：也适合纳入，但优先级低于知乎

CSDN 的优势在于：

- 网页公开内容多
- 技术文章和知识库场景很贴合
- 对 Markdown 输出、代码块、表格、公式等结构有实际价值

现成信号：

- `WeChat Article to Markdown` 已经把 CSDN 列为兼容站点
- `zhihu-download` 的更新日志也明确加入了 CSDN 下载功能

结论：

- CSDN 适合做“通用网页文章归档”的一部分
- 但平台差异价值没有微信、知乎那么高

### 4. 已经有成熟“通用 Web Clipper”，所以不要重复做所有通用网页

官方 Obsidian Web Clipper 已经能：

- 抓取网页
- 生成持久化 Markdown
- 使用模板、变量、规则做站点定制
- 本地保存到 Obsidian

这说明一件事：

- “通用网页保存为 Markdown”本身已经不是稀缺能力
- 你不应该把项目做成“又一个通用网页剪藏器”

你更适合做的是：

- 面向中文内容平台的“专用增强型归档器”
- 在现成 Web Clipper 做不好的地方提供更深的站点适配

## 是否有必要继续扩功能

有必要，但要限定扩展方式。

不建议的方向：

- 每个平台都做成“小红书同级深度”
- 每个平台都追求正文、评论、互动数据、OCR、作者关系、收藏夹/专栏批量抓取全覆盖

原因：

- 会迅速稀释项目边界
- 会显著增加维护成本
- 会让项目从“有特色的归档工具”变成“维护压力很大的半爬虫平台”

建议的方向：

- 把平台分成两类处理

### 类别 A：深度平台

特征：

- 值得做站点级深适配
- 有明显区别于通用剪藏器的价值

当前建议只保留：

- 小红书

理由：

- 收藏夹导出
- 单笔记保存
- OCR 价值高
- 评论价值高
- 通用剪藏器很难替代

### 类别 B：文章平台

特征：

- 主要价值是正文保存
- 评论、互动和复杂平台能力不是一期开销重点

建议纳入：

- 微信公众号文章
- 知乎文章
- CSDN 博客

统一支持的能力建议只包含：

- 标题
- 作者
- 发布时间
- 正文
- 图片
- 链接
- 标签/专栏名（能拿到则保留）
- Markdown 导出

默认不承诺：

- 评论
- 互动数据
- 登录后私域能力
- 平台内部收藏夹/专栏批量深爬

## 推荐方案

推荐走“**双轨扩展**”：

### 轨道 1：小红书继续做深

继续强化：

- 收藏夹工作流
- 评论提炼
- OCR 质量
- AI 摘要
- 多入口统一

### 轨道 2：增加文章型平台适配器

新增一个更轻量的“文章抓取模式”：

- 输入 URL
- 按站点适配器提取正文
- 统一转成项目自己的 note/article 对象
- 复用现有 Markdown 落地能力

这比“给每个平台都复制一份小红书逻辑”更健康。

## 建议的功能边界

### 一期建议支持

- 微信公众号文章正文保存
- 知乎文章正文保存
- CSDN 博客正文保存
- 统一 Markdown 输出
- 统一 frontmatter
- 统一图片下载策略

### 一期明确不做

- 微信公众号评论抓取
- 微信公众号互动数据抓取
- 全平台评论统一抽取
- 全平台批量账号级抓取
- 平台发布回写

## 对项目定位的影响

如果按推荐路线走，项目定位将变成：

“一个以小红书为深度主场、同时支持微信公众号/知乎/CSDN 等文章平台归档的中文内容采集与 Markdown 归档引擎。”

这个定位比“只做小红书”更宽，也比“什么都抓”更稳。

## 最终建议

结论不是“别扩”，而是“有选择地扩”。

我建议：

1. 保持小红书为深度主场
2. 新增一个轻量的文章平台模式
3. 第一批只接微信公众号文章、知乎文章、CSDN 博客
4. 微信评论不要承诺，知乎/CSDN 评论也先不做成基线能力

这样做的收益最大：

- 你自己会更常用
- 别人会更容易理解和接受
- 未来卖的时候，产品故事也更完整
