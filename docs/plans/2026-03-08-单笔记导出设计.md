# 单笔记导出设计

日期：2026-03-08

## 背景

当前项目的主链路围绕“小红书收藏夹导出”展开，但用户实际经常需要保存一条未收藏的笔记。外部实现的主流做法也很一致：接受分享文本、短链或当前笔记 URL，保留登录态后直接打开详情页并导出 Markdown，而不是要求先收藏。

本轮设计目标不是重做整套抓取架构，而是把现有的单笔记能力收敛成一条正式入口，并明确它与收藏夹导出的关系。

## 外部参考

以下资料证明“单笔记独立入口”是合理且常见的产品形态：

- `bnchiang96/xiaohongshu-importer`
  - 支持粘贴分享文本或 URL 导入
  - 兼容 `explore` 和 `discovery/item` 链接
  - 最终输出 Markdown
- 小红书官方 deeplink 文档
  - `item/<note_id>` 与 `board/<board_id>` 明确是不同路由
  - 单笔记与专辑/收藏在产品概念上本来就是并列入口
- `iFurySt/RedNote-MCP`
  - 先登录并持久化 Cookie，再按 URL 抓取单笔记
  - 说明业内常见路径是“保留登录态 + 按 URL 访问笔记”
- MediaCrawler 的 XHS 方案
  - 最终仍保留详情页/HTML 解析兜底
  - 说明详情页解析虽然脆弱，但仍是常见主路径

## 目标

- 将“单笔记导出”固化为独立入口，而不是收藏夹模式的附属分支
- 支持 `URL + 分享文本 + 短链 + --current` 四类输入
- 尽量复用现有输入规范化、详情提取、OCR、摘要、标签、Markdown 导出能力
- 保持单笔记与收藏夹/专辑导出共享底层能力，但入口独立
- 把后续评论采集、AI 处理作为附加链路接入，不反向影响主抓取结果

## 非目标

- 本轮不引入新的浏览器运行时或第二套抓取框架
- 本轮不切换到持久化 Cookie 登录方案
- 本轮不重构收藏夹/专辑导出主链路
- 本轮不做批量 URL 导入、队列调度、增量同步
- 本轮不尝试一次性解决仓库全部编码问题

## 方案对比

### 方案 A：单笔记作为独立入口，复用当前已登录 Chrome/CDP

做法：

- 保留 `scripts/save_note.js` 作为独立入口
- 输入统一规范化为 `noteId + canonical URL`
- 通过当前已登录 Chrome 打开详情页并抽取内容
- 复用现有导出链路输出 Markdown、图片、OCR、评论归档

优点：

- 与现有仓库最贴合，改动最小
- 复用率最高，不需要新增运行时依赖
- 产品边界清晰，和收藏夹/专辑模式并列

缺点：

- 依赖本机 Chrome 远程调试端口
- 环境准备比“持久化 Cookie”方案更手工

结论：推荐作为当前版本。

### 方案 B：单笔记独立入口，但改为持久化 Cookie 登录

做法：

- 引入持久化登录态
- 后续抓取不再依赖当前 Chrome 标签页

优点：

- 使用体验更稳
- 更适合自动化

缺点：

- 需要新依赖、新配置和登录态管理
- 这轮范围会明显膨胀

结论：可作为后续增强，不放进当前版本。

### 方案 C：把单笔记继续塞回收藏夹主脚本

做法：

- 在 `scripts/extract_v4.js` 或现有收藏夹脚本里继续追加 `--url` / `--current` 分支

优点：

- 表面上入口少一个

缺点：

- 职责继续膨胀
- 单笔记输入模型与收藏夹遍历模型不同，后续维护会更重

结论：不推荐。

## 推荐设计

### 产品边界

“单笔记导出”和“收藏夹/专辑导出”是两条并列入口：

- 单笔记导出
  - 面向一条明确的笔记输入
  - 允许分享文本、短链、URL、当前页
- 收藏夹/专辑导出
  - 面向一组笔记来源
  - 负责枚举出一批 note URL 或 note 对象

二者共享底层解析与导出能力，但不共享入口编排逻辑。

### 模块边界

本轮优先复用，不再新增大块抽象：

- `scripts/save_note.js`
  - 单笔记入口编排
  - 参数解析
  - 错误格式化与运行提示
- `scripts/lib/note_input.js`
  - 从 URL / 分享文本抽取链接
  - 解析 `noteId`
  - 规范化为 canonical item URL
- `scripts/lib/cdp_note.js`
  - 连接 Chrome
  - 导航详情页
  - 提取正文、媒体、评论
  - 将抓取结果映射为统一 note 结构
- `scripts/lib/note_export.js`
  - 下载媒体
  - OCR / 摘要 / 标签
  - Markdown 渲染
  - 评论归档和有用评论输出

其中：

- 直接复用：`note_input.js`、`save_note.js`、`note_export.js`
- 只补洞不重写：`cdp_note.js`
- 这轮先不碰：Cookie 持久化、Playwright 生产入口、收藏夹链路重构

### 数据流

单笔记的标准数据流如下：

1. 输入层接收 `分享文本 / 短链 / 标准 URL / --current`
2. 输入规范化层统一生成 `noteId + canonical URL`
3. 抓取层复用当前登录态，打开详情页并提取正文、媒体、评论
4. 导出层统一生成 Markdown、图片归档、评论 JSON
5. AI 后处理只作为导出附加步骤，不影响原始抓取成功与否

收藏夹/专辑后续如要复用，只需要把它们的输出收敛到同样的单 note 处理链。

## 输入设计

支持以下调用方式：

```bash
node scripts/save_note.js https://www.xiaohongshu.com/explore/<noteId>
node scripts/save_note.js "74 我刚刚在小红书看到这篇笔记，快来看！ https://xhslink.com/..."
node scripts/save_note.js --current
```

规范化原则：

- 能提取 `noteId` 时，统一构造 `https://www.xiaohongshu.com/discovery/item/<noteId>`
- 对短链和分享文本先提取 URL，再做重定向或 `noteId` 归一
- `--current` 只校验当前页，不做模糊猜测
- 非法输入直接报错

## 抓取与导出设计

### 抓取策略

URL 或分享文本模式：

- 连接本地 Chrome CDP
- 导航到规范化后的详情页
- 等页面稳定后提取字段

`--current` 模式：

- 获取当前标签页 URL
- 校验是否是详情页
- 是则直接抽取，否则报错

### note 结构

单笔记结果继续尽量贴合现有 note 结构，至少包含：

- `title`
- `content`
- `author`
- `authorLink`
- `date`
- `tags`
- `images`
- `noteUrl`
- `noteId`
- `collection`
- `accountKey`
- `accountUid`
- `accountNickname`
- `comments`

其中：

- `collection` 固定为 `单挑保存`
- `comments` 默认为空数组或采集结果

### 输出策略

单笔记固定输出到：

```text
output/{accountKey}/单挑保存/
```

继续复用现有导出链路：

1. 下载图片
2. 执行 OCR
3. 清洗文本并做可选 AI 后处理
4. 生成摘要与标签
5. 输出 Markdown
6. 如有评论则额外输出评论归档和评论区段落

重复保存同一 `noteId` 时默认覆盖重生成。

## 错误处理

需要清晰区分：

- 输入无法解析
- Chrome CDP 未启动
- 当前没有可用的小红书标签页
- 当前未登录导致详情页不可访问
- `--current` 页面不是详情页
- 页面结构变化导致字段或评论提取失败
- 图片、OCR、AI 后处理失败

处理原则：

- 入口和抓取失败时直接退出，并返回可执行的中文提示
- 评论失败不阻塞正文导出
- 图片、OCR、AI 失败时尽量保留 Markdown 主结果
- 环境问题和代码问题在输出上明确区分

## 测试边界

本轮优先验证最值钱的部分，不做重型自动化：

- 纯函数和薄封装单元测试
  - 输入规范化
  - note 映射
  - 导出路径
  - Markdown 渲染
  - 错误格式化
- 评论的结构化输出和筛选规则测试
- 浏览器侧只做小范围真实 DOM 验证

不在本轮引入：

- 持久化登录态自动化测试
- 第二套浏览器抓取入口
- 收藏夹与单笔记统一 CLI 重构

## 结论

单笔记导出应作为独立入口长期保留，收藏夹/专辑导出作为并行入口存在。当前版本最稳妥的落地方式，是继续沿用 `scripts/save_note.js + note_input.js + cdp_note.js + note_export.js` 这条链路，只在评论采集、错误提示和输入兼容上做最小补强，而不是重新造一套新框架。
