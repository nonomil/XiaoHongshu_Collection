# 项目优化路线设计

日期：2026-03-12

## 背景

当前仓库已经同时覆盖以下能力：

- 收藏夹批量抓取
- 单笔记链接/分享口令保存
- Markdown 落地
- OCR 识别
- 评论提取、筛选与归档
- AI 摘要和标签
- 本地 UI 入口

这说明项目已经越过“验证可行性”的阶段，正在从脚本集合过渡到可长期使用的本地归档工具。

用户目标不是单纯做一个自用脚本，也不是立刻做重型产品，而是同时满足三件事：

1. 自己长期稳定使用
2. 别人拿来可以用
3. 未来具备包装和售卖的可能

因此，后续优化不应继续无序堆功能，而应优先做结构收敛、稳定性提升和产品边界清晰化。

## 目标

将当前项目从“功能较多的脚本仓库”优化为“一个稳定的小红书内容归档引擎 + 多种使用入口”。

优化完成后，项目应具备以下特征：

- 核心流程统一，不同入口不再各自分叉实现
- 失败可定位、可回退、可重试
- 输出结构稳定，适合知识库长期积累
- OCR / AI / 评论处理可替换，不绑死单一实现
- 入口层可以继续扩展，而不牵动核心逻辑

## 非目标

本轮优化不追求：

- 立刻做完整商业化授权系统
- 立刻做 SaaS 或在线多人服务
- 立刻做重型前端应用
- 覆盖所有社交平台

本轮只聚焦把“小红书内容归档引擎”打磨到稳定、清晰、可扩展。

## 方案比较

### 方案 A：继续按入口演进

做法：

- 收藏夹入口继续改收藏夹脚本
- 单笔记入口继续改单笔记脚本
- UI 再包一层自己的逻辑

优点：

- 短期改动最少
- 临时需求上手快

缺点：

- 同一能力在多个入口重复实现
- 配置、错误处理、日志会越来越分裂
- 后续产品化成本高

结论：

不推荐。它适合短期迭代，不适合当前阶段。

### 方案 B：直接产品化重构

做法：

- 先重做 UI
- 先做安装器、配置界面、产品交互
- 再倒逼底层改造

优点：

- 看起来更像产品
- 有利于展示和售卖

缺点：

- 会把大量时间花在外壳而不是内核
- 核心流程还没收敛时，产品化外层会反复返工

结论：

暂不推荐。过早产品化会放大技术债。

### 方案 C：双层架构（推荐）

做法：

- 先收敛“归档引擎”
- 再让 CLI / bat / 本地 UI 复用同一套内核
- 未来需要时再叠加桌面端、服务端、商业包装

优点：

- 不耽误自己继续使用
- 也给别人提供更稳定的入口
- 将来更容易卖，因为底层已经具备产品形态

缺点：

- 现在需要先做一轮结构整理

结论：

推荐采用。它最符合当前目标和阶段。

## 推荐架构

项目后续建议收敛为四层：

### 1. 采集层

职责：

- 收藏夹抓取
- 单笔记详情抓取
- 评论展开与采集
- 账号识别
- 登录态检查

要求：

- 只负责“拿到结构化原始数据”
- 不负责 OCR、摘要、Markdown 输出

### 2. 处理层

职责：

- OCR
- OCR 清洗
- 评论去重、筛选、总结
- 标签清洗
- 摘要生成

要求：

- 输入统一的 note 对象
- 输出富化后的 note 对象
- provider 化，允许替换 OCR/AI 实现

### 3. 输出层

职责：

- Markdown 生成
- 原图下载
- 评论归档 JSON 输出
- frontmatter 规范化
- 文件命名、目录命名、去重

要求：

- 输出 schema 固定
- 同一笔记在不同入口下输出应一致

### 4. 入口层

职责：

- CLI
- bat 启动器
- 本地 UI
- 未来可扩展桌面端或服务端

要求：

- 不持有复杂业务逻辑
- 只负责收集输入、调用管线、展示结果

## 核心数据流

建议统一成一条标准管线：

`input -> fetch -> enrich -> write -> report`

对应含义：

- `input`
  - 收藏夹任务
  - 单链接任务
  - 当前页面任务
- `fetch`
  - 通过 CDP 获取结构化 note 数据
- `enrich`
  - OCR、评论处理、摘要、标签
- `write`
  - Markdown、图片、评论 JSON 落盘
- `report`
  - 返回成功数、失败数、失败原因、输出路径

这样一来，入口差异只体现在 `input`，核心流程全部复用。

## 关键优化点

### 1. 统一任务模型

建议定义统一的任务对象，例如：

- `task.type`
- `task.source`
- `task.options`
- `task.account`

并统一结果对象，例如：

- `status`
- `noteId`
- `filepath`
- `commentArchivePath`
- `warnings`
- `error`

这会直接减少入口之间的行为偏差。

### 2. 统一配置系统

建议把以下配置集中管理：

- Chrome/CDP 配置
- 输出目录
- OCR provider 配置
- AI provider 配置
- 评论策略配置
- 重试/超时策略

统一配置后，项目才适合给别人使用，也更接近可售卖产品。

### 3. 错误分级和日志

至少区分：

- 抓取失败
- 登录态失效
- 评论采集失败
- OCR 失败
- AI 失败
- 文件写入失败

并明确每类错误的退化策略：

- 评论失败不阻塞正文导出
- AI 失败回退到本地启发式
- Vision OCR 失败回退到 Tesseract

### 4. 输出 schema 固定化

现在项目最值得做成护城河的，是长期可用的归档结果。

建议固定：

- note 原始字段
- note 富化字段
- 评论归档 JSON 结构
- Markdown frontmatter 字段
- 运行报告结构

这会直接提升知识库长期可维护性。

### 5. OCR / AI provider 化

当前已经存在：

- Vision OCR
- Tesseract fallback
- OpenRouter 摘要/标签

下一步应把这些抽象成 provider 接口，而不是继续在流程里硬编码条件分支。

### 6. 评论价值提炼

评论能力应从“能抓”升级成“有用”。

建议后续重点提升：

- 线程归并
- 去重
- 作者回复优先
- 工具名、链接、纠错类评论优先
- 评论总结与正文总结分离

## 风险与应对

### 风险 1：边重构边损伤现有可用性

应对：

- 保持现有入口可跑
- 先抽象共享逻辑，再替换入口调用
- 用现有测试和样例数据做回归

### 风险 2：产品化心态过早，反而拖慢内核

应对：

- UI 只做薄壳
- 优先做引擎，不先做重前端

### 风险 3：配置与编码问题持续扩散

应对：

- 尽快统一配置
- 尽快治理编码问题
- 在测试里补上对中文文本和输出内容的保护

## 测试策略

后续优化应重点覆盖：

- 任务对象和输出对象的结构测试
- 评论筛选和 OCR 回退行为测试
- 配置缺失、配置非法时的失败路径测试
- Markdown 输出快照测试
- 多入口共用同一核心流程的集成测试

## 优先级建议

### P0：先把内核做稳

- 统一任务管线
- 统一配置系统
- 错误分级和日志
- 输出 schema 固定化
- 编码问题治理

### P1：把“能用”升级成“好用”

- 评论价值提炼
- OCR provider 化
- 运行结果汇总报告
- 更适合知识库的输出体验

### P2：为可卖做准备

- 安装和环境自检
- 更友好的入口层
- 授权/版本策略预留
- 可服务化的核心引擎

## 结论

这个项目下一阶段最值得做的不是继续横向加新功能，而是把现有能力收敛成一个稳定的归档引擎。

推荐路线是：

- 先做双层架构
- 先做稳定性和结构统一
- 再做体验升级
- 最后再逐步加产品化包装

如果这条路线执行到位，项目会同时满足三种身份：

- 自己稳定使用的私人工具
- 别人能上手的开箱工具
- 未来可包装和售卖的内容归档产品
