# 链接保存与双入口 UI 设计

## 背景

当前仓库已经具备两条能力链路：

- 收藏夹抓取：`scripts/extract_v4.js` -> `scripts/ocr_and_write.js`
- 单条笔记保存：`scripts/save_note.js`，支持分享文本、单条链接和 `--current`

用户当前需求不是继续扩展收藏夹逻辑，而是把“单条笔记保存”收敛成主项目中的正式入口，并进一步支持从一段文字中提取多条小红书链接，按顺序保存多篇笔记。同时，主项目需要一个足够轻量的本地 UI，把两类入口明确区分出来。

## 目标

- 把单条笔记输出目录名称统一为 `单条笔记保存`
- 让链接入口支持“粘贴一段文本，自动抽取其中的多条小红书笔记链接并顺序保存”
- 提供一个极简本地 UI，包含两个入口：`链接保存笔记` 与 `收藏保存笔记`
- 尽量复用现有脚本，不重造收藏抓取、OCR、导出链路

## 非目标

- 不做在线部署，不引入 React、Vue 或数据库
- 不重写收藏夹抓取逻辑
- 不做复杂任务队列、断点续跑或历史任务列表
- 不改变当前 CDP + 已登录浏览器的抓取前提

## 用户流程

### 1. 链接保存笔记

1. 用户启动本地 UI。
2. 在“链接保存笔记”区域粘贴分享文本、多个 URL 或混合文本。
3. 系统自动提取文本中的所有小红书笔记链接，去重后按出现顺序逐条保存。
4. 每条笔记仍复用 `save_note.js` 现有能力：短链处理、页面加载、评论抓取、OCR、Markdown 导出。
5. UI 返回保存总数、成功条数、失败条数和每条结果的输出路径或错误。

### 2. 收藏保存笔记

1. 用户点击“收藏保存笔记”。
2. UI 后端顺序执行现有收藏抓取脚本：`scripts/extract_v4.js` 与 `scripts/ocr_and_write.js`。
3. UI 返回执行结果与关键日志。

## 架构决策

### 输入解析

在 `scripts/lib/note_input.js` 增加批量提取能力：

- `extractUrlsFromText(input)`：从任意文本中提取所有小红书 URL
- `normalizeNoteInputs(input)`：对多条 URL 做去重与规范化

保留原有单条接口，避免影响现有调用。

### 批量保存执行器

在 `scripts/save_note.js` 中把“解析输入”和“保存一条笔记”拆开：

- 单条保存逻辑继续保留
- 新增批量模式：对 `normalizeNoteInputs()` 的结果顺序执行保存
- CLI 和 UI 统一复用这套顺序执行逻辑

这样可以避免 UI 再次重复拼装单条保存流程。

### 本地 UI

采用最小实现：

- 后端：原生 Node `http` 服务
- 前端：静态 `HTML + CSS + JS`

原因：

- 用户明确要求“简单一点”
- 现有项目没有前端框架依赖
- 该 UI 只是本地脚本入口，不值得引入额外构建系统

### API 约定

- `POST /api/save-links`
  - 输入：`{ text: string }`
  - 输出：批量保存结果，包括总数、成功数、失败数、每条详情
- `POST /api/save-collection`
  - 输入：空对象即可
  - 输出：收藏夹导出执行结果与日志

### 并发控制

本地 UI 仅允许一次运行一个任务。

原因：

- 当前项目底层依赖同一个浏览器登录态和统一输出目录
- 收藏抓取和单条保存并发执行时，容易互相污染浏览器状态和日志

因此后端会加一个简单的运行锁；若已有任务在跑，后续请求直接返回冲突错误。
