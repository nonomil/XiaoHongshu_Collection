# 统一入口、本地/云端部署与同步方案知识库

日期：2026-04-07

## 1. 这份知识库解决什么问题

当前项目已经从“小红书单一导出脚本”演进为“多来源链接归档执行器”：

- 小红书：本地 Chrome/CDP 抓取后落盘 Markdown
- 微信公众号 / 知乎 / CSDN：可直接按 URL 保存为 Markdown
- Pushbullet：已接入为外部收件箱入口

现在真正需要统一的，不是再加一个零散入口，而是把下面几件事分清楚：

1. 哪些工具适合当“入口”
2. 哪些工具适合当“执行器”
3. 哪些适合云端部署
4. 哪些必须优先留在本地
5. 本地与云端如何同步，不互相打架

---

## 2. 先给结论

### 2.1 最适合当前项目的总体架构

建议采用三层结构：

1. 入口层：负责把链接、页面、消息送进系统
2. 队列/收件箱层：负责缓冲、去重、记录来源和路由
3. 执行器层：负责真正抓取、增强、写 Markdown

当前项目最合适的分工是：

- 本项目主仓库：继续担任执行器内核
- Pushbullet / 浏览器插件 / 飞书 / 未来 IMA CLI：担任入口
- 云端服务：优先担任统一入口与任务队列，不强行承担全部执行

### 2.2 本地优先与云端优先的边界

本地优先：

- 小红书单条保存
- 需要登录态的浏览器抓取
- 依赖当前浏览器上下文、人工介入、评论预热的任务

云端优先：

- 链接收集
- 公众号文章、知乎文章、CSDN 文章等纯 URL 后处理
- Webhook、机器人、浏览器插件的统一入站
- 定时同步、批量重试、监控和通知

### 2.3 当前最推荐的路线

Phase 1：

- 修好 Pushbullet 全量导入与稳定增量
- 统一收件箱模型
- 增加按年/月归档的本地落盘方式

Phase 2：

- 增加统一 ingress API
- 接浏览器插件、本地 UI、Pushbullet、飞书 webhook
- 云端先做“接收并入队”

Phase 3：

- 云端 worker 处理公众号 / 知乎 / CSDN
- 小红书继续本地执行，云端只收链接或做辅助调度

## 3. 当前项目现状

### 3.1 已有能力

仓库当前已经具备这些基础能力：

- 统一任务管线：`input -> fetch -> enrich -> write -> report`
- 统一任务模型：`task`
- 收件箱链路：`Pushbullet -> inbox_links.jsonl -> saveInboxUrls`
- 多来源识别：小红书、微信公众号、知乎文章/回答、CSDN
- 本地 UI 和 CLI 双入口

### 3.2 当前真实短板

短板不在“没有入口”，而在“入口与执行器没有彻底解耦”：

- Pushbullet 目前只有一个来源实现
- 外部入口尚未统一为 ingress 协议
- 收件箱还偏向单文件 JSONL，没有分层沉淀
- 云端部署设计已有草案，但还没真正落地

## 4. 方案分类图

### 4.1 按角色分类

#### A. 入口型方案

特点：

- 负责收链接、收页面、收消息
- 不承担复杂抓取
- 可本地，可云端

包括：

- Pushbullet
- 浏览器插件
- 飞书 webhook / 机器人
- OpenClaw / ClawCloud 的聊天入口
- 手机分享桥接
- 未来 IMA CLI

#### B. 执行器型方案

特点：

- 真正负责抓正文、图像、评论、OCR、AI 增强、Markdown 落盘
- 对浏览器上下文和登录态更敏感

当前最适合继续承担执行器的就是本项目本身。

#### C. 混合型方案

特点：

- 既能作为入口，也能承担一部分后处理
- 但通常不适合替代主执行器

包括：

- Wechatsync / 文章同步助手
- 云端 worker

## 5. 各方案逐项评估

### 5.1 Pushbullet

定位：

- 最适合做“轻量收件箱入口”

优点：

- 你已经在用，接入成本最低
- API 目前还能拉取完整历史
- 适合手机到桌面的快速投递

缺点：

- 网页端不适合做长期总库
- 不是正文提取器
- 当前仓库实现默认分页偏保守，容易误以为历史丢失

适合本地：

- 很适合

适合云端：

- 适合当云端入站源

不适合：

- 不适合直接当最终知识库

结论：

- 保留
- 修复全量同步
- 从“唯一入口”降级为“入口之一”

---

### 5.2 浏览器插件

定位：

- 最适合做“一键发送当前页到本地/云端”

优点：

- 用户体验最直接
- 最适合替代“打开 bat + 复制粘贴”
- 本地和云端都可接

缺点：

- 不应承载抓取内核
- 不适合直接在插件里实现小红书采集逻辑

适合本地：

- 非常适合

适合云端：

- 适合作为云端统一入口

结论：

- 高优先级
- 作为统一 ingress 的最佳前端入口

---

### 5.3 飞书

定位：

- 最适合做“团队消息入口、Webhook 入队、通知回传”

优点：

- 官方开放平台能力成熟
- 适合消息触发、机器人、审批和结果通知
- 比较适合云端接入

缺点：

- 不是公众号正文抓取器
- 不适合替代本项目的小红书执行链路

适合本地：

- 可以接，但收益一般

适合云端：

- 很适合

结论：

- 飞书更像“中控台 / 控制面”
- 适合收链接、下命令、回结果
- 不建议把它当主内容采集器

---

### 5.4 OpenClaw / ClawCloud

说明：

- 这里把你提到的“云端龙虾”先按 OpenClaw / ClawCloud 这一类聊天网关来理解

定位：

- 最适合做“云端聊天入口聚合器”

优点：

- 能接多聊天渠道
- Feishu 通道成熟
- 支持本地默认地址，也支持远程访问
- 适合作为云端控制入口和任务分发入口

缺点：

- 它不是小红书抓取执行器
- 它不负责 CDP、评论采集、Markdown 落盘主逻辑

适合本地：

- 可用，但价值主要在多端消息入口

适合云端：

- 非常适合

结论：

- 很适合当“云端总入口”
- 不适合直接替换当前项目主执行器

---

### 5.5 Wechatsync / 文章同步助手

定位：

- 最适合做“长文类来源的 Markdown 补充工具”

优点：

- 更擅长公众号、通用网页文章、Markdown 导出
- 更适合浏览器文章转 Markdown
- 对长文内容整理更友好

缺点：

- 不是小红书抓取器
- 不应替代当前项目的统一任务模型

适合本地：

- 适合

适合云端：

- 更适合作为补充工具，不建议当主云端执行框架

结论：

- 适合作为“公众号 / 长文转 Markdown 的旁路工具”
- 不适合作为整个系统的主中枢

---

### 5.6 IMA CLI

定位：

- 潜在的企业文档/知识库入口适配器

当前状态：

- 本仓库还没有现成实现
- 是否适合主线，要看它能否稳定输出结构化文档或 Markdown

适合本地：

- 待验证

适合云端：

- 如果 CLI 无浏览器强依赖，则很适合云端 worker

结论：

- 适合做成可插拔入口适配层
- 不宜先把主线架构绑死在它身上

## 6. 本地部署、云端部署、混合部署

### 6.1 本地部署

最适合当前项目现状。

原因：

- 小红书抓取依赖本地 Chrome/CDP
- 登录态复用已经围绕本机构建
- 评论抓取、人工介入、验证码处理都更适合本地

适合承载：

- 小红书执行器
- 单篇保存
- 本地 UI
- 项目专用浏览器 profile

### 6.2 纯云端部署

当前不适合直接把整个项目原样搬上云端后宣称“全部可跑”。

原因：

- 小红书对登录态、环境、风控、评论加载非常敏感
- 本地当前浏览器模式无法直接照搬到云端
- 云端浏览器一旦被风控，维护成本会迅速上升

适合承载：

- 统一 ingress API
- webhook
- 队列
- 公众号 / 知乎 / CSDN 的 URL 后处理
- 定时任务与重试任务

### 6.3 混合部署

这是最推荐的最终形态。

建议分工：

- 本地：
  - 小红书执行器
  - 高风险、强登录态任务
- 云端：
  - 统一入口
  - 队列与任务编排
  - 公众号/知乎/CSDN 的稳定后处理
  - 飞书 / OpenClaw / Pushbullet / webhook 接入

---

## 7. 本地与云端如何同步

### 7.1 最推荐的同步模型

不要做“本地 output 目录和云端 output 目录互相硬同步”。

更合适的是：

1. 同步任务，而不是同步整个输出目录
2. 同步结果元数据，而不是直接双向覆盖 Markdown
3. 把原始收件箱、任务状态、结果索引分开存

### 7.2 推荐的数据分层

#### 原始入口层

例如：

- `data/inbox_links.jsonl`
- `data/inbox_archive/YYYY/YYYY-MM.jsonl`

用途：

- 保存所有入口事件
- 支持重放和补跑

#### 任务状态层

例如：

- `data/tasks/*.json`
- 或未来换 SQLite

用途：

- 记录是否已执行
- 记录在哪个环境执行
- 记录失败原因和重试次数

#### 结果索引层

例如：

- `data/output_index/YYYY/YYYY-MM.json`

用途：

- 记录 URL 与本地 Markdown 的映射
- 便于云端/本地查询，不必扫描全部输出目录

#### 内容输出层

例如：

- `output/收件箱同步/2026/2026-04/...`
- `output/{账号}/...`

用途：

- 保留最终 Markdown、评论归档、图片

### 7.3 同步原则

原则 1：

- 入口事件只追加，不覆盖

原则 2：

- 任务状态由执行端负责更新

原则 3：

- 最终 Markdown 只在“指定主写入端”生成

原则 4：

- 本地与云端不要同时处理同一条小红书任务，避免竞态和重复落盘

## 8. 归档策略建议

### 8.1 收件箱层

保留两个视角：

1. `实时收件箱`
2. `历史归档桶`

建议：

- `data/inbox_links.jsonl` 只作为近期队列和事实流
- 已处理或已沉淀内容按年/月归档到 `data/inbox_archive/`

### 8.2 Markdown 输出层

建议新增年月分层：

```text
output/
  收件箱同步/
    2026/
      2026-04/
        微信公众号文章/
        知乎文章/
        知乎回答/
        小红书/
        CSDN/
      全部/
```

其中：

- `全部/` 保留全集视图
- 年月目录作为日常查看主入口

### 8.3 为什么不要只保留“全部”

因为随着数量增长，会出现：

- 查找困难
- 补跑成本高
- 同标题碰撞增多
- 云端/本地排错困难

## 10. 最终推荐方案

### 10.1 你现在最值得做的，不是再选一个单点工具

而是先把架构定成：

- 本项目 = 执行器
- Pushbullet / 插件 / 飞书 / OpenClaw / IMA CLI = 入口
- 云端 = 统一入站 + 队列 + 可云端处理的 worker

### 10.2 最终推荐组合

本地：

- 当前项目主仓库
- Chrome/CDP
- 小红书执行

云端：

- ingress API
- webhook
- OpenClaw / Feishu 入口
- 公众号/知乎/CSDN worker

外部入口：

- Pushbullet：轻量投递
- 浏览器插件：一键发送
- 飞书：消息与任务控制
- Wechatsync：长文类补充导出
- IMA CLI：待验证后作为企业知识入口适配器

### 10.3 一句话版判断

- 想抓小红书：优先本地
- 想统一多端入口：优先云端 ingress
- 想接聊天工具：优先飞书 / OpenClaw
- 想处理公众号长文：可引入 Wechatsync 作为补充
- 想长期稳定：一定要把入口、队列、执行器拆开

## 12. 当前知识库结论状态

已确认：

- Pushbullet API 可拉历史，不应以网页端显示为准
- 本项目适合继续做执行器
- 小红书当前应以本地执行为主
- 飞书 / OpenClaw 更适合做入口和控制面
- 云端最适合先做 ingress、队列和可后处理来源

待后续补证：

- IMA CLI 的真实能力边界
- Wechatsync 是否值得直接纳入主线自动化，而不是旁路工具
- 云端任务状态层是否继续用 JSONL/JSON，还是升级 SQLite
