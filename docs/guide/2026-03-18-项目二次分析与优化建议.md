# 2026-03-18 项目二次分析与优化建议

日期：2026-03-18

## 1. 当前项目定位

当前仓库最合理的定位，不是“通用中文内容爬取平台”，而是：

- 以小红书为深度主场的本地内容归档引擎
- 以 Markdown / 本地知识库落地为最终输出
- 以 URL / 收件箱为扩展入口，而不是把所有平台都做成同等深度

从代码现状看，项目已经具备以下骨架：

- 统一任务对象：`scripts/lib/task.js`
- 统一执行管线：`scripts/lib/pipeline.js`
- 统一导出能力：`scripts/lib/note_export.js`
- OCR provider 层：`scripts/lib/ocr_provider.js`
- Summary provider 层：`scripts/lib/summary_provider.js`
- UI 薄壳化：`scripts/ui_server.js` + `ui/app.js`

这说明项目已经走出“脚本堆叠期”，正在进入“内核收敛期”。

## 2. 当前最大结构问题

### 2.1 单笔记与收藏夹仍然是两套主链路

单笔记保存已经较好地走进共享内核：

- `scripts/save_note.js`
- `scripts/lib/task.js`
- `scripts/lib/pipeline.js`
- `scripts/lib/note_export.js`

但收藏夹导出仍保留旧式分叉流程：

- `scripts/extract_v4.js`：抓取收藏夹并写入 `data/raw_notes.json`
- `scripts/ocr_and_write.js`：再读原始 JSON 做 OCR / 摘要 / Markdown
- `scripts/ui_server.js`：UI 端仍通过 spawn 两个脚本串行执行

这会带来三个直接问题：

- 共享能力无法完整复用，收藏夹链路持续偏离主内核
- UI 层仍然需要理解“双脚本流程”，不是纯薄壳
- 后续 Pushbullet / IFTTT / 飞书 / 手工 URL 都难以自然复用“同一套保存能力”

### 2.2 收藏夹入口过于硬编码

`scripts/extract_v4.js` 当前把收藏夹板块写死在 `BOARDS` 常量中。

这限制了：

- 多账号使用
- 新收藏夹接入
- 别人开箱即用
- 后续 UI 配置化

### 2.3 文档存在路线分叉

当前仓库中同时存在两类路线：

- 路线 A：继续只做小红书，把小红书做深
- 路线 B：扩展到公众号 / 知乎 / CSDN 等文章平台

从代码实际落地看，仓库目前明显更接近路线 A。

如果不先冻结边界，后续会在“做深小红书”和“横向扩平台”之间反复摇摆。

## 3. 对外部来源的重新判断

### 3.1 Pushbullet

适合作为第一外部入口保留。

原因：

- 当前已有增量拉取、分页与统一配置
- 路径短，能真实支撑日常手机到桌面的收集流程
- 能自然进入“收件箱 -> 保存”的统一链路

### 3.2 IFTTT

适合作为下一阶段入口胶水，不适合先做成重逻辑节点。

更合理的位置是：

- 作为 webhook 转发器
- 把稍后读、浏览器收藏、RSS、邮件等来源统一推入 inbox

### 3.3 飞书

更适合作为通知 / 人工审核 / 轻交互层，而不是第一优先级采集层。

更建议后续用于：

- 导出完成通知
- 失败任务提醒
- 人工确认分类

### 3.4 OpenClaw

如果后续接入，更适合作为远程控制层，而不是采集 provider。

更合理的用途是：

- 远程触发同步
- 查询最近一次运行结果
- 执行失败重跑

### 3.5 公众号 / 知乎 / CSDN / 通用网页

这些平台不是当前仓库的第一优先级。

原因不是“没有价值”，而是：

- 当前仓库最大的技术债并不在平台数量，而在主链路尚未彻底收敛
- 小红书已经具备明显差异化深度，扩平台会稀释主场价值
- 通用网页保存本身已经有成熟 Web Clipper 产品

因此更稳妥的策略是：

- 当前仓库先固定为“小红书深度主场 + 通用 URL 入口”
- 真要扩“文章平台模式”，建议独立模块化甚至新仓库

## 4. 推荐优化路线

### P0：先收敛主链路

优先做：

- 收藏夹导出并入共享管线
- 收藏夹抓取与写入解耦
- UI 收藏导出改为直连共享服务
- 保留现有 CLI / UI 入口兼容

### P1：再稳定执行与恢复

优先做：

- 每次运行输出结构化 report
- 为失败项保留重试信息
- 为每条笔记保留更稳定的 result shape
- 让“只重跑失败项”成为可能

### P2：最后扩入口

优先做：

- InboxItem schema 固定化
- Pushbullet 之外的 provider 接口统一
- IFTTT / 飞书 / 手工粘贴 / 收藏网址统一进 inbox

## 5. 本轮建议落地主题

本轮最值得先做的一件事是：

**把收藏夹导出从“双脚本旧流程”收敛为“共享收藏导出服务 + 兼容入口”。**

原因：

- 这是当前结构分叉最大的地方
- 这是后续 inbox / Pushbullet / IFTTT / 飞书复用的前置条件
- 这比继续扩新平台更能直接提升项目可维护性和产品化基础

## 6. 一句话结论

当前仓库下一步不该继续横向扩平台，而应该先把“收藏夹导出”真正拉回统一内核。

正确顺序应是：

`先统一收藏夹主链路 -> 再稳固 report / retry / inbox -> 最后再扩外部入口与新平台`
