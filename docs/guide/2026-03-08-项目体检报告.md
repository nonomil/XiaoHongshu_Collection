# 项目体检报告

日期：2026-03-08

## 1. 项目概览

本项目是一个面向 Windows 环境的小红书收藏夹导出工具，目标是把收藏夹中的笔记抓取到本地，并进一步整理为结构化 Markdown 文档。

当前主流程分为两段：

1. 抓取阶段
   - 通过 Chrome CDP 连接已经登录的小红书页面
   - 模拟点击收藏夹中的笔记卡片
   - 从页面 DOM 中提取标题、正文、作者、日期、标签、图片链接
   - 写入 `data/raw_notes.json`
2. 整理阶段
   - 下载图片并执行 OCR
   - 对 OCR 文本做清洗和可选 AI 纠错
   - 对笔记生成摘要和标签
   - 输出 Markdown 到 `output/{账号_key}/{收藏夹}/`

从仓库现状看，当前的实际主入口是：

- `scripts/extract_v4.js`
- `scripts/ocr_and_write.js`

## 2. 当前目录职责

- `scripts/`
  - 抓取、OCR、导出主流程脚本
- `scripts/ai/`
  - 可单测的辅助模块，承担账号识别、摘要解析、OCR 后处理、标签清洗等职责
- `config/`
  - OpenRouter 配置
- `data/`
  - 抓取阶段输出的中间数据，目前核心文件是 `raw_notes.json`
- `output/`
  - 最终 Markdown 和图片输出目录
- `docs/`
  - 使用说明、历史方案和技能文档
- `assets/`
  - Tesseract 训练数据等静态资源

## 3. 技术实现判断

### 3.1 优点

- 主链路清晰，抓取与整理已经拆成两个阶段，便于单独调试
- 已经开始从“大脚本”向“模块化辅助函数”演进
- `scripts/ai/__tests__/` 下已有 Node 原生测试
- 当前测试可运行，`node --test scripts/ai/__tests__/*.test.js` 在本地通过
- README 已经覆盖基本启动步骤、配置说明和输出目录结构

### 3.2 当前成熟度

整体处于“可用脚本项目”阶段，不是“工程化应用”阶段。

特征如下：

- 有明确主流程
- 有部分测试
- 模块拆分刚起步
- 依然依赖手工运行脚本和本地环境约束
- 缺少统一命令入口、持续校验和错误恢复机制

## 4. 核心数据流

### 4.1 抓取链路

1. 连接 `http://localhost:9222/json`
2. 找到小红书标签页 WebSocket 地址
3. 启用 CDP `Network` 和 `Runtime`
4. 在收藏夹页遍历卡片
5. 通过鼠标事件或 DOM click 打开笔记
6. 从当前页面提取内容
7. 落盘到 `data/raw_notes.json`

### 4.2 导出链路

1. 读取 `data/raw_notes.json`
2. 以账号维度构建输出目录
3. 下载笔记图片
4. 调用 Tesseract 做 `chi_sim+eng` OCR
5. 清洗 OCR 文本，并按阈值决定是否调用 AI 做纠错
6. 调用 AI 生成摘要与标签，失败时回退到本地逻辑
7. 输出 Markdown

## 5. 主要发现

### 5.1 编码污染是当前第一风险

仓库内有多处中文乱码痕迹，尤其是以下文件：

- `data/raw_notes.json`
- `scripts/write_markdown.js`
- `docs/guide/xhs-collection-export.md`
- `scripts/ai/account.js` 中的“关注”后缀清理逻辑也出现乱码形式

这不是显示层问题，而是数据层和源码层都可能已经被错误编码写入。其影响包括：

- 抓取后的原始数据不可逆污染
- Markdown 导出内容异常
- 文本清洗规则失效
- 标签、作者名、收藏夹名判断错误
- 后续 AI 输入质量下降

这是最优先需要确认和收敛的问题。

### 5.2 页面结构依赖较强

`scripts/extract_v4.js` 依赖以下不稳定因素：

- 小红书页面 DOM 结构
- class 命名片段
- 卡片渲染时机
- 固定等待时间
- 点击后详情页的打开方式

这类脚本通常在页面改版后脆弱。当前实现已经有重试和降级逻辑，但仍属于高耦合抓取方案。

### 5.3 配置和运行入口不统一

`package.json` 当前只有依赖，没有脚本入口，例如：

- 没有 `npm test`
- 没有 `npm run extract`
- 没有 `npm run ocr`

这会带来三个问题：

- 新人接手门槛高
- 命令约定分散在 README 中
- 后续自动化校验难接入

### 5.4 新旧脚本存在职责重叠

`scripts/write_markdown.js` 和 `scripts/ocr_and_write.js` 存在重叠导出职责，但前者明显更旧，且已出现编码污染。

如果继续同时保留两套导出路径，风险是：

- 修一个流程漏另一个流程
- 输出格式不一致
- 文档和实际行为逐渐偏离

### 5.5 仓库当前处于未清理工作区

当前 git 工作区不是干净状态，存在已修改和未跟踪文件，包括：

- `README.md`
- `scripts/extract_v4.js`
- `scripts/ocr_and_write.js`
- 多个 `scripts/ai/*.js`
- 多个 `scripts/ai/__tests__/*.test.js`
- `assets/tesseract/*.traineddata`

这说明项目正在演进中。后续任何重构都要先区分：

- 哪些是当前有效改动
- 哪些是历史遗留
- 哪些只是本地资源文件

## 6. 当前可维护性评估

### 6.1 可维护部分

- `scripts/ai/summary.js`
- `scripts/ai/account_detect.js`
- `scripts/ai/tesseract_path.js`
- `scripts/ai/__tests__/`

这部分已经具备“小模块 + 小测试”的维护条件。

### 6.2 维护成本较高部分

- `scripts/extract_v4.js`
- `scripts/ocr_and_write.js`

原因是：

- 文件偏长
- 同时承担流程控制、数据转换、错误处理、文件系统写入、第三方调用
- 对外部系统依赖较多

这两个文件适合作为下一轮拆分重点。

## 7. 建议的优先级

### P0：先解决编码问题

目标：

- 确认哪些文件源文本已被污染
- 区分“终端显示乱码”与“文件真实内容乱码”
- 重新校准中文常量、正则和测试样例

建议动作：

1. 抽样检查源码文件的实际编码
2. 确认 `raw_notes.json` 是否还能从源流程重新生成
3. 统一脚本文件保存为 UTF-8
4. 修正涉及中文匹配的规则和测试

### P1：明确唯一主流程

目标：

- 固定当前推荐使用的抓取脚本和导出脚本
- 标记旧脚本状态

建议动作：

1. 将 `extract_v4.js` 和 `ocr_and_write.js` 定义为唯一主流程
2. 将旧脚本补充为“废弃”或迁移到 `legacy/`
3. 在 README 中只保留当前主路径

### P2：补齐工程化入口

目标：

- 降低运行和维护门槛

建议动作：

1. 在 `package.json` 中补充 `test`、`extract`、`ocr` 等脚本
2. 增加基础校验命令
3. 明确训练数据目录约定

### P3：拆分长脚本

目标：

- 降低主脚本复杂度

建议动作：

1. 从 `extract_v4.js` 抽出 CDP 通信层、DOM 提取层、抓取调度层
2. 从 `ocr_and_write.js` 抽出配置加载、图片下载、OCR 服务、Markdown 渲染层
3. 为关键纯函数补测试

## 8. 推荐的后续执行顺序

建议按以下顺序推进，而不是同时大改：

1. 编码体检与修复
2. 清理旧脚本与文档入口
3. 统一 `package.json` 命令
4. 拆分 `extract_v4.js`
5. 拆分 `ocr_and_write.js`
6. 最后再考虑更强的异常恢复、增量同步和批量处理能力

## 9. 结论

这个项目已经具备“能跑通真实业务链路”的基础，价值点明确，主流程也基本成形。当前最大的短板不是功能缺失，而是工程质量不稳定，特别是编码污染、长脚本耦合和入口分散。

如果只做一件事，应该优先处理编码问题。编码问题不收敛，后续无论是 OCR 质量、AI 摘要、标签清理，还是 Markdown 输出，都会持续产生脏数据。

在编码问题确认后，这个仓库适合继续沿着“脚本主流程 + 可测试辅助模块”的方向逐步重构，而不是一次性改造成复杂应用。
