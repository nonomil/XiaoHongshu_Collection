# Pushbullet 最近 50 条双库落盘验证

Date: 2026-03-23

## 验证目标

确认当前项目在 Pushbullet 收件箱场景下：

1. 能同步最近 50 条消息
2. 能保存其中的知乎与微信公众号文章
3. 会同时写入：
   - 分类笔记库
   - 总笔记库 `收件箱同步/全部`
4. 同标题不同链接不会在总库中互相覆盖

## 本次执行

### 1. 最近 50 条同步

执行 `syncInbox({ mode: 'recent', limit: 50 })` 后，返回结果：

- `mode`: `recent`
- `limit`: `50`
- `total`: `50`
- `added`: `0`
- `skipped`: `50`
- `truncated`: `true`
- `warning`: `Pushbullet pull capped at maxItems=50.`

说明：

- 这 50 条消息已经在本地收件箱存储中存在
- 本次同步主要用于“取最近 50 条做保存验证”，不是新增入库验证

### 2. 最近 50 条中的目标来源

从最近 50 条里筛出：

- 微信公众号链接：10 条
- 知乎回答 / 专栏链接：5 条
- 合计：15 条

### 3. 实际保存验证

对最近 50 条中的知乎与微信公众号候选链接进行实跑和落盘反查后，结果为：

- 15 / 15 条都至少存在 2 份落盘副本
- 每条都能在输出库中找到：
  - 1 份分类稿
  - 1 份总库镜像稿

代表性样例：

- 微信公众号总库镜像：

  - `output/收件箱同步/学习/怎么把知乎里的好答案原封不动的保存下来？.md`
- 知乎总库镜像：
  - `output/收件箱同步/全部/怎么把知乎里的好答案原封不动的保存下来？.md`

## 发现的问题

### 同标题不同链接会覆盖总库镜像

在验证过程中发现一个真实问题：

- 两篇不同知乎回答标题相同
- 分类目录各自正常保存
- 但总库 `收件箱同步/全部` 原先使用默认覆盖策略
- 会导致后保存的文件覆盖先保存的文件

复现场景：

- `https://www.zhihu.com/question/1999136031413384196/answer/2008749278114361561`
- `https://www.zhihu.com/question/1999136031413384196/answer/2009032881960424982`

它们标题都为：

- `codex可以100%正式接管所有编程工作了吗？`

## 已修复

修复方式：

- 收件箱保存默认启用 `content-aware` 冲突策略
- 当总库中出现“同标题但正文不同”时，自动生成后缀文件

修复后结果：

- 分类稿：
  - `output/收件箱同步/工具/codex可以100%正式接管所有编程工作了吗？.md`
- 总库镜像保留第一份：
  - `output/收件箱同步/全部/codex可以100%正式接管所有编程工作了吗？.md`
- 总库镜像保留第二份：
  - `output/收件箱同步/全部/codex可以100%正式接管所有编程工作了吗？-1.md`

说明当前行为已经符合预期：

- 分类库保留分类视图
- 总库保留完整归档
- 同标题不同内容不会再互相覆盖

## 当前结论

当前收件箱流程在“最近 50 条消息 -> 知乎 / 公众号文章保存 -> 分类库 + 总库双份落盘”这条链路上已经验证通过。

本轮验证覆盖结论：

- Pushbullet 最近 50 条同步可用
- 微信公众号文章保存可用
- 知乎回答 / 专栏保存可用
- 分类笔记库与总笔记库双份落盘可用
- 同标题不同链接的总库覆盖问题已修复

## 下一步建议

最值得继续做的不是再扩展保存数量，而是补一层“验证报告自动化”：

1. 在 CLI 中固化“最近 N 条双库核对”命令，避免每次用内联脚本手查
2. 在 UI 里给收件箱同步结果增加“已保存 / 未保存 / 已双份镜像”统计
3. 对小红书短链也做同样的最近 N 条回查报告，便于发现评论采集失败与镜像缺失

## 已落地的自动化核对

上述第 1 项现在已经落地为仓库命令：

```powershell
npm run inbox:sync -- --mode recent --limit 50
npm run inbox:verify -- --limit 50
```

对应脚本：

- `scripts/inbox_sync.js`
- `scripts/inbox_verify_recent.js`
- `scripts/lib/inbox_verify.js`

当前这条核对链路的作用是：

- 直接从 `data/inbox_links.jsonl` 的最近 N 条里筛公众号 / 知乎链接
- 扫描 `output/收件箱同步/**/*.md`
- 输出每个目标 URL 是否同时具备：
  - 分类稿
  - 总库镜像稿
