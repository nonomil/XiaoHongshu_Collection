# 知乎收藏夹全量保存调研

Date: 2026-03-22

## 问题

当前项目已经支持：

- 知乎专栏文章单篇保存
- 知乎回答页单篇保存
- 通过统一 `save_note.js` 流程写入 Markdown、图片、摘要和标签

但还不支持“给一个知乎收藏夹 URL，把收藏夹里的全部条目完整导出到本地”的批量能力。

这份调研要回答两个问题：

1. 当前仓库距离“知乎收藏夹全量保存”还差什么
2. 最适合当前项目的落地方式是什么

## 当前项目已有基础

### 已具备的能力

- URL 识别已经支持 `zhihu_article` 与 `zhihu_answer`
  - `scripts/lib/source_detector.js`
- 单篇解析器已经存在
  - `scripts/lib/sources/zhihu.js`
- 单篇保存已经接入统一主流程
  - `scripts/save_note.js`
- 输出模型已经支持知乎平台元数据
  - `scripts/lib/note_export.js`

### 当前短板

- 还没有“知乎收藏夹 URL -> 条目列表”的抓取器
- 还没有收藏夹批量导出的进度持久化
- 还没有知乎收藏夹场景下的节流 / 403 退避 / 单篇容错
- 还没有“按收藏夹组织输出目录”的专用落盘规则

结论很明确：当前仓库已经具备“单条条目落盘”能力，但缺少“收藏夹目录层”的抓取与编排层。

## 参考方案里最值得借鉴的点

本地参考文档 `tools/simpread-怎么把知乎里的好答案原封不动的保存下来？.md` 和开源项目 `download-zhihu` 指向了一个很清楚的方向：不要靠页面滚动硬抓收藏夹列表，而是直接走收藏夹分页接口，把“目录抓取”和“单篇导出”分成两层。

### 1. 收藏夹目录不要依赖滚动懒加载

参考项目的关键做法是直接分页请求收藏夹内容 API，而不是模拟页面一直往下滚。

这对当前仓库非常重要，因为我们已经在小红书和公众号场景里验证过一件事：

- 页面滚动适合拿单篇详情
- 列表型批量任务更适合做“接口层编排”

### 2. 导出进度必须持久化

参考项目把进度写成 `export-progress-{id}.json` 这类文件，中断后可继续。

这很适合当前仓库，因为我们已经有：

- 本地输出目录
- 任务型 UI
- 用户长期运行、睡觉前开任务的真实场景

如果不加进度持久化，知乎收藏夹导出在收藏数达到几百条时会很脆弱。

### 3. 要有单篇失败容错，而不是一条失败整批中断

参考项目在收藏夹导出里保留了单篇失败日志和继续执行能力。

这和当前仓库的方向完全一致。我们在收件箱同步、小红书批量保存里已经证明：

- 允许单条失败
- 汇总 `successCount` / `failureCount`
- 保留失败项重试入口

这套产品模型可以直接复用到知乎收藏夹导出。

### 4. 长文内容补全不能只信第一页 DOM

参考项目后续加了 `initialData + DOM` 的双源补全，用于解决长文截断。

当前仓库的知乎单篇解析器还偏轻，只做了：

- DOM 文本抽取
- 页面侧 `Runtime.evaluate`

这说明知乎收藏夹全量导出如果要做稳定，不能只把当前的 `scripts/lib/sources/zhihu.js` 直接原样批量复用，还需要补一个“长文补全层”。

### 5. 需要节流和 403 退避

参考项目显式加入了请求节流和 403 指数退避。

这也非常契合当前项目，因为：

- 知乎收藏夹是明显的批量读取场景
- 当前仓库已经在小红书评论采集里遇到过频率和风控问题
- 批量收藏夹任务如果不降速，很容易出现半程失败

## 对当前项目最合适的实现路线

### 总原则

不要把知乎收藏夹做成一个独立的全新系统。

更适合当前项目的方案是：

- 新增“收藏夹目录抓取层”
- 目录层拿到全部条目 URL
- 单篇条目继续复用现有 `save_note.js` 和 `processSingleNoteExport()`

也就是：

`知乎收藏夹 URL -> 收藏夹条目列表 -> 逐条复用现有单篇保存 pipeline`

这条路线的好处是：

- 和当前项目架构一致
- 不会分叉出第二套 Markdown 导出逻辑
- UI、日志、失败重试、分类、标签能力都能复用

## 推荐的分阶段落地

### 第一阶段：先打通“目录抓取 + 逐条保存”

建议新增：

- `scripts/lib/zhihu_favorites.js`
- `scripts/save_zhihu_favorites.js`
- `scripts/ai/__tests__/zhihu_favorites.test.js`

第一阶段只做这些：

- 输入一个知乎收藏夹 URL
- 解析收藏夹 ID 和标题
- 分页抓取收藏夹条目 URL
- 为每个条目生成统一任务对象
- 调用现有单篇保存链路
- 把文件写到 `output/知乎收藏夹/<收藏夹标题>/`
- 生成进度文件 `output/知乎收藏夹/<收藏夹标题>/_state/export-progress-<id>.json`

### 第二阶段：补长文补全与风控策略

这一阶段再增加：

- `initialData + DOM` 双源提取
- 收藏夹分页节流
- 403 / 429 退避
- 失败条目日志与二次重试
- 已导出条目去重

### 第三阶段：接入 UI

等 CLI 稳定后，再加一个首页入口或设置项：

- 直接粘贴知乎收藏夹 URL 导出
- 或在“外部收件箱”里识别知乎收藏夹链接后转专用流程

不建议一开始就把 UI 和抓取器同时上线。

## 具体实现建议

### 1. 目录抓取层

建议在 `scripts/lib/zhihu_favorites.js` 里拆成三类函数：

- `parseFavoriteMeta(url, html)`
- `fetchFavoriteItems({ collectionId, offset, limit, cookies })`
- `collectFavoriteEntries({ url, browserSession, progress })`

其中最关键的是 `collectFavoriteEntries()`，它负责：

- 复用当前登录态
- 分页拉取条目
- 标准化为统一条目对象

### 2. 登录态策略

优先级建议：

1. 复用当前 Chrome 146+ 已登录会话
2. 复用项目自己的持久 profile
3. 允许手工传 Cookie 作为兜底

原因：

- 知乎收藏夹很多内容只对登录用户可见
- 当前项目已经在浏览器接管与项目 profile 复用上做了不少工作
- 继续沿用这条主线，风险最小

### 3. 输出组织

建议输出为两层：

- 主目录：`output/知乎收藏夹/<收藏夹标题>/`
- 镜像目录：继续允许按分类落到 `知乎文章` / `知乎回答`

这样用户既能看到“这个收藏夹的完整备份”，也不丢掉现有的按来源或分类浏览方式。

### 4. 与现有保存链路的衔接

对每一条收藏夹条目，目录层不直接写 Markdown，而是只负责组装：

- `input`
- `canonicalUrl`
- `sourceType`
- `collectionResolver`
- `mirrorTargets`

然后交给现有 `save_note.js`。

这样能直接复用：

- AI 摘要与标签
- 图片下载
- 命名冲突策略
- UI 报告结构
- 失败重试机制

## 风险

### 1. 知乎接口和 DOM 都可能变化

所以目录层和正文层要解耦：

- 目录层尽量依赖分页接口
- 正文层保留 DOM 与页面评估双兜底

### 2. 收藏夹里可能混有失效、付费或受限内容

需要把这类条目标成 warning，而不是让整批中断。

### 3. 长时间任务容易中断

所以进度文件不是可选项，而是基线能力。

## 结论

当前项目做“知乎收藏夹全部保存”是可行的，而且不需要另起炉灶。

最合理的做法不是模仿浏览器扩展去重写整套导出器，而是：

1. 新增收藏夹目录抓取层
2. 用分页 API 拉完整条目列表
3. 逐条复用现有单篇保存 pipeline
4. 补进度持久化、节流、长文补全和失败续跑

如果只看工程性价比，第一阶段最值得先做的是：

- 收藏夹 URL 解析
- 收藏夹条目分页抓取
- 逐条保存
- `export-progress-{id}.json`

这四项落地后，知乎收藏夹导出就已经能进入“可用但仍需继续打磨”的状态。
